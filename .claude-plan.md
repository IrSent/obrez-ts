# Plan: Fix audio buffer overlap (multiple segments playing simultaneously)

## Root Cause

**Primary: `onended` fires before audio truly stops on the render thread.**

Web Audio processes in render quanta (128 samples ≈ 2.67ms at 48kHz). The `onended` event fires when the SourceNode reaches its stop time, but the node is still producing audio in the current quantum. The next buffer is scheduled to start at `actualEndCorrection = ctx.currentTime` (from `onended`), which is ~1 quantum BEFORE the previous buffer is truly silent → two buffers overlap for ~2.67ms per transition.

At >1x: **0ms margin** — overlap is guaranteed.
At 1x: **0.5ms margin** — insufficient (need ~3ms).

This means **every single buffer boundary** during normal playback has a small overlap, causing audible clicks/pops and the sensation of "multiple segments playing."

**Secondary: `stopAudio()` → `startAudio()` gap is tight.**

After `stopAudio()` calls `node.stop(0)` on all queued nodes and waits 10ms, `startAudio()` creates new nodes. 10ms ≈ 4 quanta — usually enough, but if any node was scheduled for the future (bootstrap, gap filler), it may not have started yet and `stop(0)` cancels it silently. The 10ms wait is a heuristic, not a guarantee.

**Tertiary: `finally` block releases lock before `onended` fires.**

When `iterator.return()` is called, the for-await loop breaks, the `finally` block fires immediately (releasing `runAudioIteratorLockRef`), but the last BufferSource's `onended` hasn't fired yet. `stopAudio()` sees the lock released and proceeds to stop all nodes — but some nodes that were just created in the last iteration haven't finished producing audio.

## Fixes

### Fix 1: Add render-quantum margin to `actualEndCorrection` (main fix)

**File:** `useMediaPlayer.ts`, lines ~909-915

The `onended` handler sets `actualEndCorrection = ctx.currentTime`. This is the time when `onended` fires, NOT when the node is truly silent. Add a margin of ~3ms (1 render quantum at 48kHz) to ensure the next buffer starts only after the previous one is truly done.

```typescript
// Before:
if (actualEndCorrection != null && currentSpeed > 1) {
  lastEnd = actualEndCorrection;
  actualEndCorrection = null;
} else if (actualEndCorrection != null && currentSpeed === 1) {
  lastEnd = actualEndCorrection + 0.0005;
  actualEndCorrection = null;
}

// After:
const RENDER_QUANTUM_MARGIN = 0.003; // ~128 samples at 48kHz
if (actualEndCorrection != null) {
  lastEnd = actualEndCorrection + RENDER_QUANTUM_MARGIN;
  actualEndCorrection = null;
}
```

This eliminates the per-boundary overlap at ALL speeds.

### Fix 2: Wait for all `onended` before releasing the lock

**File:** `useMediaPlayer.ts`, `runAudioIterator` finally block (~line 1024-1026)

Currently the `finally` block releases the lock immediately. We need to wait for all queued BufferSource nodes to truly finish (their `onended` to fire and the render quantum to complete) before releasing the lock.

Approach: in the `finally` block, wait for `queuedAudioNodesRef.current.size === 0` (all `onended` have fired), then add an extra 3ms margin.

```typescript
finally {
  // Wait for all currently-queued nodes to fire onended before releasing the lock.
  // This ensures stopAudio (which waits for the lock) doesn't proceed until
  // the last buffer has truly finished producing audio.
  if (queuedAudioNodesRef.current.size > 0) {
    await new Promise((resolve) => {
      const id = setInterval(() => {
        if (queuedAudioNodesRef.current.size === 0) {
          clearInterval(id);
          resolve();
        }
      }, 10);
      // Safety: don't hang forever
      setTimeout(resolve, 500);
    });
  }
  // Extra margin: onended fired, but render thread may still be producing audio
  await new Promise(r => setTimeout(r, 3));
  runAudioIteratorLockRef.current = false;
}
```

### Fix 3: Increase `stopAudio` node-stop wait from 10ms to 20ms

**File:** `useMediaPlayer.ts`, line ~489

After `node.stop(0)` on all queued nodes, we wait 10ms before proceeding. Increase to 20ms to ensure 2+ render quanta have passed and all nodes are truly silent.

```typescript
// Before:
await new Promise(r => setTimeout(r, 10));

// After:
await new Promise(r => setTimeout(r, 20));
```

### Fix 4: Defensive — check for truly-silent before starting new buffers

**File:** `useMediaPlayer.ts`, `startAudio` (~line 547-560)

After creating the new iterator, add a small delay before the bootstrap silence to ensure no lingering audio from the previous context exists. This is a belt-and-suspenders measure — Fix 2 and Fix 3 should make this unnecessary, but it costs only ~5ms.

No change needed — Fix 2 already ensures the lock isn't released until nodes are truly silent.

## Summary of changes

| Fix | Location | Impact |
|-----|----------|--------|
| 1 | `onended` → `actualEndCorrection` margin | Eliminates per-boundary overlap (main cause) |
| 2 | `finally` block waits for `onended` | Ensures lock release = truly silent |
| 3 | `stopAudio` 10ms → 20ms | Extra safety for stop→start transition |
